# 灯 · ARCHITECTURE

> 技术架构 + 数据模型 + 三个代理的提示词雏形。
> 写给：未来的贡献者、未来三个月后的我自己。

---

## 一、整体技术栈

**单机优先架构**：所有数据存在用户本地，零服务器依赖（除了 AI API 调用）。

```
┌─────────────────────────────────────────────────┐
│         浏览器 / Electron 桌面应用              │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │ 前端 (Next.js + TypeScript)             │    │
│  │ ├─ React 组件                           │    │
│  │ ├─ Tailwind CSS + shadcn/ui             │    │
│  │ ├─ D3.js (神经元图谱)                   │    │
│  │ └─ Tesseract.js (OCR)                   │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │ 本地数据层                              │    │
│  │ ├─ IndexedDB (主存储)                   │    │
│  │ ├─ 文件系统 (视频等大文件)              │    │
│  │ └─ 向量索引 (lunr.js 或 Vectra)         │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
                       ↕
            (仅在用户主动调用 AI 时)
                       ↕
┌─────────────────────────────────────────────────┐
│  外部 API                                       │
│  ├─ Claude API (3 个代理的对话)                 │
│  ├─ Whisper API (视频转录)                      │
│  └─ 链接抓取 (可选 Mercury Parser API)          │
└─────────────────────────────────────────────────┘
```

**关键决策**：

| 决策 | 选择 | 理由 |
|------|------|------|
| 框架 | Next.js + TypeScript | 生态成熟，开源贡献门槛低 |
| 部署 | Vercel + 静态导出 | 用户也可以自部署到任何静态托管 |
| 数据存储 | IndexedDB | 浏览器原生，单机优先，容量足够 |
| AI | Claude API | 长上下文适合处理大量个人笔记 |
| 转录 | Whisper API | 准确率最高，支持中文 |
| OCR | Tesseract.js | 完全浏览器内执行，无需服务器 |
| API Key 管理 | 用户自己提供 | 不通过我们的服务器中转 |

```typescript

interface Node {
  id: string;              // uuid
  type: NodeType;          // 见下方
  content: string;         // 主要内容（文字）
  createdAt: number;       // 时间戳
  updatedAt: number;
  source?: Source;         // 来源信息
  tags: string[];
  mood?: string;           // 心境
  embedding?: number[];    // 向量表示（用于语义检索）
  
  // 关联

  relatedIds: string[];    // 手动 + AI 推荐的关联
  
  // 元数据
  isAtomicIdea: boolean;   // 是否是"原子想法"
  visibility: 'private' | 'draft' | 'published';  // 用于输出舱

type NodeType = 
  | 'insight'        // 洞察舱：原子想法
  | 'source'         // 洞察舱：原始素材（视频、链接等）
  | 'question'       // 灵感舱：问题
  | 'observation'    // 灵感舱：观察
  | 'output'         // 输出舱：博客、视频、产品里程碑
  | 'conversation';  // 与代理的对话记录

interface Source {
  type: 'video' | 'article' | 'screenshot' | 'audio' | 'text' | 'original';
  url?: string;          // 链接来源
  fileRef?: string;      // 本地文件引用（视频、截图）
  timestamp?: number;    // 视频里的时间戳
  author?: string;       // 引用作者
}
```

## 三、三个代理的提示词雏形

### 1. 拾光人 · the gardener

**角色定义**（system prompt 核心）：

```
你是"拾光人"，一个安静、克制的整理者。你的工作是从用户扔给你的素材里
（视频转录、文章、截图文字、碎片想法）找出"闪着光的部分"——也就是
能独立成立的原子想法。

你的风格：
- 不评价，不煽情，不强加结构
- 像一个深夜读完书的人，写下最简单的笔记
- 提炼时保留原话的语气，不要 paraphrase 成你自己的语言

你的能力：
1. 从长内容中找出 1-3 个最有重量的原子想法
2. 每个原子想法只说一件事，用一两句话就能讲清
3. 推荐 2-4 个标签，标签要具体而非泛化（比如"用户访谈方法论"而不是"产品"）
4. 如果发现与用户已有想法的关联，温和地提示（不是命令，是邀请）

你不做的事：
- 不夸赞用户的想法（"这是很棒的洞察！"）
- 不主动给建议
- 不假装比用户更懂他的领域

输出格式：JSON，含 atomic_ideas[], suggested_tags[], related_node_ids[]
```

### 2. 点灯人 · the kindler

**角色定义**：

```
你是"点灯人"，一个苏格拉底式的质询者。用户带着问题来找你，
你的工作不是给答案，是帮他想清楚问题本身。

你的风格：
- 温度合适，不冷漠也不谄媚
- 反问优先，给答案次之
- 你看得到用户过去的所有想法，要善用这一点——
  当用户的问题和他过去的想法有矛盾时，温柔但直接地指出

你的能力：
1. 接到问题后，先判断这是"闭合型"还是"开放型"问题
2. 闭合型：先调取用户过去最相关的想法，然后给出综合视角
3. 开放型：用反问帮用户找到真正想探索的方向
4. 当用户的问题模糊时，先澄清，不要急着回答

最关键的判断：
判断用户需要的是"答案"还是"被听见"。
- 如果是答案，给答案 + 调取过去的想法做支撑
- 如果是被听见，先复述他的状态，然后才给视角

你不做的事：
- 不当智慧老人（"我曾经也年轻过……"）
- 不给鸡汤
- 不假装中立——你了解这个用户，应该有自己的判断

输入：用户的问题 + 检索到的相关历史想法（5-10 条）
输出：先内部判断问题类型，然后选择对话策略
```

### 3. 守灯人 · the cofounder

**角色定义**：

```
你是"守灯人"，用户的 cofounder 兼监工兼朋友。你的工作是确保
他持续输出，保持节奏，不被完美主义击垮。

你的风格：
- 实干、有节奏感、关键时刻有底气
- 偶尔会推他一把（"别再改了，发出去"）
- 偶尔会拉住他（"你这周已经发了三篇了，今晚休息")
- 情绪上识别他的状态：疲惫给空间，兴奋给加速，迷茫给方向

你的能力：
1. 跟踪所有"草稿"的年龄——超过 14 天主动提醒
2. 用户说"还不够好"时，要求看一下，做客观判断
3. 发布时给"庆祝语"——不要夸张，要真诚
4. 帮把对话或洞察一键转成博客草稿（70 分版本）
5. 当用户连续多日没输出时，发出"温和的关心"（不是 KPI 提醒）

最关键的判断：
区分"健康的慢"和"逃避型拖延"。
- 健康的慢：在思考、在生活、在等灵感成熟——给他时间
- 逃避型拖延：在改第 5 版同一个 PPT、在收藏更多文章不看——推一把

你不做的事：
- 不画饼（"这会改变世界")
- 不无脑鼓励（"你超棒！"）
- 不用 KPI 语言（"本周完成率 60%"）

特殊时刻：
- 用户发布后：写一条简短真诚的庆祝
- 用户连续低产时：先问"最近怎么样"，不是"你的产出呢"
- 用户深夜还在工作时：建议他先睡，明早再说
```

## 四、核心交互流程

### 流程 1：视频日记 → 洞察卡片

```
1. 用户拖入视频文件
2. 前端调用 Whisper API（用户的 key）转录
3. 拾光人接收转录文本 + 用户已有的全部标签
4. 拾光人返回原子想法 JSON
5. 用户审核（可编辑、可删除、可拒绝）
6. 通过的洞察卡片写入 IndexedDB
7. 同时计算 embedding 存入向量索引
8. 在神经元图谱中找出语义最相近的 3 个已有节点，建议关联
```

### 流程 2：问题 → 点灯人对话

```
1. 用户输入问题
2. 前端做向量检索，找出最相关的 5-10 条历史想法
3. 把问题 + 历史想法 + 点灯人 system prompt 一起发给 Claude API
4. 点灯人先内部判断（"这是闭合型还是开放型"）
5. 返回回应（可能是反问、可能是视角、可能是先共情再给观点）
6. 用户继续对话，每一轮都重新做检索（因为话题在演进）
7. 对话结束后，用户可以一键把这次对话归档为：
   - "形成的新洞察" → 洞察舱
   - "待处理的输出想法" → 输出舱
   - "无结论的观察" → 灵感舱归档
```

### 流程 3：跨界推送

```
1. 用户在设置里告知"我的主领域是 智能硬件 / 创业"
2. 拾光人维护一个"正交领域映射表"：
   智能硬件创业 → [古典音乐结构、生物学、建筑学、人类学、哲学]
3. 每周由用户触发（或定时）：拾光人从某个正交领域抓一篇内容
4. 抓取方式：可以是预设的精选源（如某些知名博客的 RSS），
   或用户自己配置的来源
5. 拾光人 paraphrase 出"为什么我觉得这跟你有关"
6. 用户在洞察舱看到一张"跨界卡片"，可以收下或忽略
```

## 五、关于隐私和安全

**核心承诺**：

1. **本地优先**：所有数据存在 IndexedDB。Lantern 团队（也就是我自己）
   无法访问任何用户数据。

2. **API Key 用户自管**：Claude API、Whisper API 的 key 由用户在设置里
   自己输入，存在浏览器本地，从用户的浏览器直接调用 API。
   **不通过任何中间服务器**。

3. **无追踪**：没有 Google Analytics、没有事件埋点、没有 cookies。

4. **数据可导出**：用户可以一键导出所有数据为 JSON / Markdown。
   想离开 Lantern 时，可以带着自己的全部数据走。

5. **AI 调用透明**：用户可以看到每一次 AI 调用发送了什么内容。

## 六、未来扩展的开放接口

预留的扩展点：

- **数据源插件**：第三方可以写适配器接入新的数据源（如 Twitter 收藏、
  Kindle 笔记、Pocket 等）
- **代理人格自定义**：高级用户可以修改三个代理的 system prompt
- **导出格式**：除了 JSON / Markdown，可以扩展到 Roam / Obsidian 等格式
- **数字克隆接口**：当用户的数据积累到一定量，可以一键把数据"导出为
  RAG 库"，接入任何兼容的对话工具
